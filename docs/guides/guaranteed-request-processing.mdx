---
title: Guaranteed Request Processing (GRP)
description: Build resilient integrations using GRP to handle transient failures gracefully.
---



Guaranteed Request Processing (GRP) is a resilience mechanism that queues requests automatically when a downstream service is temporarily unavailable, instead of failing immediately. When GRP is enabled and a service is unreachable, the API returns **HTTP 202 Accepted** with a request ID — and processes the request asynchronously once the service recovers.

:::tip[When to use GRP]
Enable GRP for critical operations (card creation, card linking, card blocking) where you'd rather wait for eventual completion than handle a transient failure and retry yourself.
:::

---

## How It Works

1. **Send a request** with a GRP-enabled API key
2. If the downstream service is available, you get the normal response immediately
3. If the service is temporarily unavailable, you receive a **202 Accepted** response with an `x-request-id` header
4. The API queues your request and retries automatically
5. Once processed, you receive a `request.completed` webhook event
6. You can also poll for status at any time using the request ID

```
Normal flow:
  Request ──► Service Available ──► 200/201 Response

GRP flow:
  Request ──► Service Unavailable ──► 202 Accepted (x-request-id)
                                          │
                                          ▼
                                    Queued & Retried
                                          │
                                          ▼
                                    request.completed webhook
```

## Enabling GRP

GRP is controlled per API key. It is **disabled by default**.

### Via the Dashboard

1. Navigate to **Settings → API Keys**
2. Create a new key or edit an existing one
3. Toggle **Guaranteed Request Processing** on
4. Save

### Via the API

When creating or updating a secret key, set `grpEnabled: true`:

```bash
POST /secret-keys
Authorization: Bearer sk.live.xxxxx

{
  "name": "Production Key (GRP)",
  "scopes": ["cards:create", "cards:update:status", "cards:update:link"],
  "grpEnabled": true
}
```

GRP works with both secret keys (`sk.live.*`, `sk.test.*`) and machine keys.

## The 202 Response

When GRP intercepts a service-unavailable error, the response looks like this:

```http
HTTP/1.1 202 Accepted
x-request-id: req_abc123xyz789
Content-Type: application/json

{
  "code": "request-accepted",
  "message": "Request accepted and will be processed asynchronously",
  "error": {
    "code": "service-unavailable",
    "message": "Service temporarily unavailable"
  }
}
```

**Important:** Save the `x-request-id` value. You'll use it to check the request status or correlate it with the webhook event.

## Tracking Request Status

### List Requests

```bash
GET /requests
Authorization: Bearer sk.live.xxxxx
```

Supports standard [pagination and filtering](/guides/pagination-and-filtering/). For example, to find pending requests:

```bash
GET /requests?filter=status||EQ||pending
```

### Get a Specific Request

```bash
GET /requests/req_abc123xyz789
Authorization: Bearer sk.live.xxxxx
```

**Response:**

```json
{
  "_id": "req_abc123xyz789",
  "status": "completed",
  "path": "/cards",
  "method": "POST",
  "apiVersion": "2025-06-01",
  "attempts": [
    {
      "statusCode": 202,
      "latency": 150,
      "responseBody": {
        "code": "request-accepted",
        "error": { "code": "service-unavailable" }
      },
      "createdAt": "2025-03-31T10:15:00Z"
    },
    {
      "statusCode": 201,
      "latency": 2340,
      "responseBody": {
        "id": "c.xxxxx",
        "status": "created"
      },
      "createdAt": "2025-03-31T10:15:30Z"
    }
  ],
  "requestBody": { },
  "source": {
    "ipAddress": "192.168.1.1"
  }
}
```

### Request Statuses

| Status | Description |
|---|---|
| `pending` | Request is queued and waiting to be processed |
| `completed` | Request has been processed (check `attempts` for the final result) |
| `cancelled` | Request was cancelled before it could be processed |

## Webhook Notification

When a GRP request completes, a `request.completed` event is sent to your registered webhooks:

```json
{
  "event": "request.completed",
  "data": {
    "_id": "req_abc123xyz789",
    "status": "completed",
    "path": "/cards",
    "method": "POST",
    "attempts": [ ... ]
  },
  "metadata": {
    "sentAt": "2025-03-31T10:15:30Z",
    "event": "evt.xxxxx"
  }
}
```

## Integration Pattern

Here's a recommended pattern for handling GRP responses:

```typescript
async function createCard(payload: CreateCardPayload) {
  const response = await fetch("https://api.allawee.com/cards", {
    method: "POST",
    headers: {
      "Authorization": `Bearer ${API_KEY}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify(payload),
  });

  if (response.status === 201) {
    // Normal response — card created immediately
    return { status: "created", data: await response.json() };
  }

  if (response.status === 202) {
    // GRP accepted — save the request ID for tracking
    const requestId = response.headers.get("x-request-id");
    await saveGRPRequest(requestId, "card-creation", payload);
    return { status: "pending", requestId };
  }

  // Other error — handle normally
  throw new Error(`Request failed: ${response.status}`);
}
```

Then handle the webhook to complete the flow:

```typescript
app.post("/webhooks/allawee", (req, res) => {
  const { event, data } = req.body;

  if (event === "request.completed") {
    const requestId = data._id;
    const lastAttempt = data.attempts[data.attempts.length - 1];

    if (lastAttempt.statusCode >= 200 && lastAttempt.statusCode < 300) {
      // Request succeeded — process the result
      completeGRPRequest(requestId, lastAttempt.responseBody);
    } else {
      // Request ultimately failed
      failGRPRequest(requestId, lastAttempt);
    }
  }

  res.sendStatus(200);
});
```

## Testing GRP

Use the simulation endpoints in test mode to trigger GRP behavior. These endpoints intentionally throw a service-unavailable error to simulate the GRP flow.

**Prerequisites:** Your test API key must have `grpEnabled: true`.

```bash
# Simulate GRP card creation
POST /simulation/grp/cards

# Simulate GRP card link via PAN
POST /simulation/grp/cards/link

# Simulate GRP card link via card ID
POST /simulation/grp/cards/:id/link

# Simulate GRP block card
POST /simulation/grp/cards/:id/block

# Simulate GRP unblock card
POST /simulation/grp/cards/:id/unblock
```

Each simulation endpoint validates the request body the same way as the real endpoint, then throws a `ServiceUnavailable` error. With GRP enabled, the `RequestExceptionInterceptor` catches this and returns a 202 response.

### Testing Headers

| Header | Purpose |
|---|---|
| `x-grp-force-retry` | Force the system to simulate a retry scenario |
| `x-grp-force-hard-stop` | Force immediate failure instead of queueing |

## Important Notes

- **GRP only handles transient failures.** Validation errors, authentication failures, and other client errors (4xx) are returned immediately regardless of GRP settings.
- **Only `service-unavailable` errors are queued.** Other server errors are not caught by GRP.
- **GRP requires API version `2024-05-01` or later.** Keys using older API versions will not trigger GRP, even if `grpEnabled` is set.
- **The `request.completed` webhook fires once** per completed request. Make sure you have a webhook registered to receive it.
- **Each attempt is recorded** in the `attempts` array, so you can see the full retry history including latency and response for each attempt.
